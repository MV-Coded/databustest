# Жизненный цикл игры

```
    POST /game/new
         │
         ▼
    ┌──────────┐
    │ WAITING  │  ← создана, ждёт второго игрока
    └─────┬────┘
          │ POST /game/{id}/join  (с правильным join_secret)
          ▼
    ┌──────────────┐
    │ IN_PROGRESS  │  ← X и O ходят по очереди
    └──┬───────────┘
       │
       ├─── X делает 3 в ряд ────┐
       ├─── O делает 3 в ряд ────┤
       ├─── Ничья (9 ходов) ─────┤
       └─── Таймаут (forfeit) ───┤
                                 ▼
                         ┌──────────┐
                         │ FINISHED │  ← winner известен
                         └─────┬────┘
                               │ через 5 минут GC удаляет
                               ▼
                          (удалена из памяти)
```

## Фаза 1: Создание (WAITING)

Игра начинается не с первого хода, а с ожидания второго игрока. Это фундаментальное Zero Trust-решение:

- Создатель получает **только свой токен** (X) и **join_secret** — одноразовый пароль для второго игрока.
- join_secret нужно передать второму игроку через **side channel** (чат, звонок, мессенджер). Это гарантирует, что только тот, кому создатель доверил секрет, сможет присоединиться.
- Попытка сделать ход в статусе `WAITING` вернёт ошибку — игра ещё не началась.

```bash
curl -s -X POST http://localhost:8000/game/new
# → { "game_id": "uuid", "player_token": "jwt...", "join_secret": "uuid" }
```

## Фаза 2: Присоединение → IN_PROGRESS

Второй игрок предъявляет join_secret. Сервер проверяет:
- Существует ли игра?
- Не присоединился ли уже кто-то?
- Правильный ли секрет?

Если всё ок — статус становится `IN_PROGRESS`, второй игрок получает свой JWT (привязанный к `sub: "O"`).

```bash
curl -s -X POST http://localhost:8000/game/{id}/join \
  -d '{"join_secret":"..."}'
# → { "player_token": "jwt..." }
```

**Важно:** join_secret — одноразовый. Повторное использование:

```bash
curl -s -X POST ...
# → 403: "Invalid join secret or game already started"
```

## Фаза 3: Игра (IN_PROGRESS)

Игроки ходят по очереди. Каждый ход проходит **6 уровней проверки**:

| Уровень | Проверка | Что проверяет сервер | Код ошибки |
|---|---|---|---|
| 0 | **Rate limit** | Не спамит ли клиент? (10 запросов/с) | `429` |
| 1 | **JWT валидация** | Не фальшивый ли токен? Не отозван ли? | `401` |
| 2 | **Привязка к game_id** | Токен от этой игры или от другой? | `403` |
| 3 | **Анти-спуфинг** | Player_id в токене совпадает с player_id в теле? | `403` |
| 4 | **Очерёдность + таймаут** | Твой сейчас ход? Оппонент не превысил 5 минут? | `403` / `408` |
| 5 | **Статус игры** | Игра ещё идёт или уже закончена? | `400` |
| 6 | **Позиция** | Клетка свободна? (0–8, Pydantic) | `400` / `422` |

Если все проверки пройдены — сервер меняет состояние доски, переключает очередь и проверяет победителя:

```python
self.board[position] = player
self.move_count += 1

if self._check_win(player):
    self.status = FINISHED
    self.winner = player
elif self.move_count >= 9:
    self.status = FINISHED  # ничья

self.current_turn = Player.O if player == Player.X else Player.X
```

## Фаза 4: Завершение (FINISHED)

Игра заканчивается в трёх случаях:
1. **Победа** — один из игроков собрал 3 в ряд (8 возможных комбинаций)
2. **Ничья** — все 9 клеток заняты, победителя нет
3. **Forfeit** — оппонент не ходил 5 минут (таймаут)

После завершения ходы больше не принимаются:

```bash
curl -s -X POST ... -d '{"player_id":"X","position":4}'
# → 400: "Game is already finished"
```

Через 5 минут финишированную игру забирает сборщик мусора (`GameGC`).

# Как делать ходы: инструкция для каждого игрока

## Игрок X (создатель)

```bash
# 1. Создать игру
GAME=$(curl -s -X POST http://localhost:8000/game/new)
ID=$(echo "$GAME" | jq -r '.game_id')
TOKEN_X=$(echo "$GAME" | jq -r '.player_token')
JOIN_SECRET=$(echo "$GAME" | jq -r '.join_secret')

# 2. Передать JOIN_SECRET второму игроку (через чат, телеграм, звонок)

# 3. Дождаться, пока второй игрок присоединится

# 4. Проверить статус
curl -s http://localhost:8000/game/$ID \
  -H "Authorization: Bearer $TOKEN_X"
# должно быть "status": "in_progress"

# 5. Сделать ход (например, в центр — позиция 4)
curl -s -X POST "http://localhost:8000/game/$ID/move" \
  -H "Authorization: Bearer $TOKEN_X" \
  -H "Content-Type: application/json" \
  -d '{"player_id":"X","position":4}'
# → current_turn станет "O"

# 6. Ждать хода O. Проверять доску можно в любой момент:
curl -s http://localhost:8000/game/$ID \
  -H "Authorization: Bearer $TOKEN_X"
# → если current_turn снова "X" — твой ход
```

## Игрок O (присоединившийся)

```bash
# 1. Получить GAME_ID и JOIN_SECRET от игрока X

# 2. Присоединиться
JOIN=$(curl -s -X POST "http://localhost:8000/game/$ID/join" \
  -H "Content-Type: application/json" \
  -d "{\"join_secret\":\"$JOIN_SECRET\"}")
TOKEN_O=$(echo "$JOIN" | jq -r '.player_token')

# 3. Дождаться своей очереди (current_turn = "O")
#    Если X уже сходил — можно ходить сразу

# 4. Сделать ход (например, в угол — позиция 0)
curl -s -X POST "http://localhost:8000/game/$ID/move" \
  -H "Authorization: Bearer $TOKEN_O" \
  -H "Content-Type: application/json" \
  -d '{"player_id":"O","position":0}'
```

## Кто когда ходит: таблица

| Номер хода | Игрок | `current_turn` после хода |
|---|---|---|
| 1 | X | `O` |
| 2 | O | `X` |
| 3 | X | `O` |
| 4 | O | `X` |
| 5 | X | `O` |
| 6 | O | `X` |
| 7 | X | `O` |
| 8 | O | `X` |
| 9 | X | игра заканчивается (если не было победы раньше) |

## Что означают позиции (0–8)

```
 0 | 1 | 2
---+---+---
 3 | 4 | 5
---+---+---
 6 | 7 | 8
```

## Что будет, если...

| Ситуация | Результат |
|---|---|
| Ход на занятую клетку | `400` "Position already occupied" |
| Ход не в свою очередь | `403` "Not your turn. It is O's turn" |
| Ход с токеном X, но `player_id: "O"` | `403` "Token is for player X..." |
| Ход с токеном от другой игры | `403` "Token does not belong to this game" |
| Ход без токена | `401` "Invalid or expired token" |
| Ход с истёкшим таймаутом | `408` "Opponent timed out" (+ forfeit) |
| 25 ходов за 1 секунду | `429` "Too many requests" |
| Позиция 99 | `422` (Pydantic: le=8) |

## Можно ли удалять созданные игры?

**Нет.** В API нет эндпоинта `DELETE /game/{id}`. Это сознательное архитектурное решение.

### Почему?

Zero Trust означает «никакому клиенту нельзя доверять». Если дать клиенту возможность удалять игры, злоумышленник может:
- Удалить чужую игру, в которой участвует
- Уничтожить доказательства (историю ходов)
- Провести атаку на доступность (удалять игры оппонента в ключевой момент)

### Как игры удаляются на самом деле

Работает автоматический **сборщик мусора (GameGC)** — поток-демон, запущенный внутри сервера:

| Условие | Через сколько | Код в `game_gc.py` |
|---|---|---|
| Игра завершена (победа/ничья) | 5 минут | `_cleanup_finished()` — проверяет `FINISHED_GAME_TTL = 300` |
| Игра идёт, но никто не ходит > 24 часов | 24 часа | `_cleanup_stalled()` — проверяет `STALLED_GAME_TIMEOUT = 86400` |
| Достигнут лимит памяти (1000 игр) | При создании | `MAX_TOTAL_GAMES = 1000` — `POST /game/new` → `429` |

GC запускается каждые 30 секунд (`CLEANUP_INTERVAL = 30.0`) и заодно чистит stale ключи rate-limiter.

### Единственный ручной способ «удалить» игру

Можно ускорить удаление, завершив игру через **forfeit** (принудительная победа по таймауту):

```bash
# Если оппонент не ходит больше 5 минут
curl -s -X POST "http://localhost:8000/game/$ID/forfeit" \
  -H "Authorization: Bearer $TOKEN_X"
# → игра переходит в FINISHED, через 5 минут её удалит GC
```

До этого — только ждать, пока GC сделает свою работу.

## Что ещё умеет сервер (помимо игры)

### Просмотр метрик

```bash
curl -sf http://localhost:8000/admin/metrics | jq .
# → {
#     "total_games": 42,        ← сколько игр в памяти
#     "waiting": 5,             ← ждут второго игрока
#     "in_progress": 20,        ← идут прямо сейчас
#     "finished": 17,           ← завершены (будут удалены через 5 мин)
#     "blacklist_size": 0       ← сколько JWT отозвано
#   }
```

### Отзыв токена (JWT blacklist)

Поскольку каждый JWT содержит уникальный `jti` (JWT ID = UUIDv4), сервер может отозвать конкретный токен. Сейчас нет публичного эндпоинта для отзыва (чтобы не давать клиентам лишних привилегий), но внутренний механизм готов:

```python
# security.py
def revoke_token(self, jti: str) -> bool:
    """Отзывает токен. После этого decode_token() вернёт None."""
    with self._lock:
        self._blacklist[jti] = datetime.utcnow().timestamp()
```

Blacklist хранится in-memory и очищается от старых записей через сутки (`cleanup_blacklist`).

## Безопасность: как Zero Trust реализован на каждом уровне

### Уровень сети (Docker)

```
host:8000  ←─── только для отладки
    │
    ├─── player_x_net:  server + player_x + attacker
    ├─── player_o_net:  server + player_o
    └─── admin_net:     server (изолирована, только метрики)
```

- player_x **физически не может** сделать ping или curl на player_o — они в разных bridge-сетях
- admin_net **никому не доступна**, кроме сервера

### Уровень аутентификации (JWT)

- Каждый токен уникален (`jti`)
- Привязан к `game_id` (нельзя использовать в другой игре)
- Привязан к `player_id` (`sub: "X"` или `"O"`)
- Имеет срок действия (24 часа по умолчанию)
- Может быть отозван через blacklist по `jti`

### Уровень приложения (FastAPI)

- Pydantic-валидация каждого входящего JSON (типы, диапазоны)
- Rate-limiting на каждый эндпоинт (разные tier, разные лимиты)
- Per-game Lock (`threading.Lock`) — race conditions невозможны
- Аудит-трейл (история ходов сохраняется в `move_history`)

### Уровень хранения

- Сервер — единственный источник истины. Клиент получает **только** snapshot (`to_response()`).
- В ответе **нет** `your_player_id` — клиент знает свою роль из JWT.
- Игры автоматически удаляются GC — ручной «мусор» не накапливается.
